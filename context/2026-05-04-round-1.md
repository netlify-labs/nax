## Defects (Detail)

### R1 — `maxTokens` silently ignored on every AI call (critical)

This is the headline issue from this review. The Vercel AI SDK was upgraded to v6 (`ai@^6.0.168` in `package.json`) which renamed `maxTokens` → `maxOutputTokens` (`node_modules/ai/dist/index.d.mts:599`). The Netlify functions still pass `maxTokens` everywhere:

- `netlify/functions/classify-reply.ts:56` — `maxTokens: 100`
- `netlify/functions/extract-ooo-date.ts:63` — `maxTokens: 50`
- `netlify/functions/summarize-thread.ts:83` — `maxTokens: 300`
- `netlify/functions/lib/prompts/outbound-workflow.ts:630` — `maxTokens: 500`
- `netlify/functions/lib/prompts/reply.ts:156` — `maxTokens: 300`
- `netlify/functions/generate-email.ts:63` — forwards `built.maxTokens`

The reason this slipped past `tsc` is the four `as Parameters<typeof generateText>[0]` casts in `netlify/functions/lib/ai-provider.ts` (lines 169, 189, 230, 249). They convert the call arguments to the SDK's exact type, bypassing the excess-property check that would otherwise have flagged `maxTokens`. AI SDK v6 ignores unknown fields, so the model just runs to its provider default cap (Anthropic, OpenAI, and Google all default to 4096+ output tokens). The visible consequences: classification calls that should be ≤100 tokens can spend ~40× that, OOO extraction calls that should be ≤50 tokens can spend ~80×, the per-tier `TIMEOUT_BY_TIER` budget (8s for classification, 15s for generation) becomes much more likely to fire, and AI Gateway spend is uncapped per-call.

### R2 — Identity webhook accepts unsigned payloads when secret is unset (warning)

`netlify/functions/identity-validate.ts:73` only verifies the webhook signature when a secret is configured. If `IDENTITY_WEBHOOK_SECRET`, `NETLIFY_JWT_SECRET`, and `JWT_SECRET` are all unset, the function happily runs the @netlify.com gating against any caller's POST body. This function is described in `AGENTS.md` and in its own header comment as the "authoritative enforcement" of signup gating, so a misconfigured stage silently downgrades the gate. The fail-open behavior is also invisible — there is no warning log, and no env-var precondition check at boot.

### R3 — `Record<string, any>` in outbound-workflow violates project rules

`netlify/functions/lib/prompts/outbound-workflow.ts:479` and `:602` use `Record<string, any>` for `accountData`, which directly contradicts AGENTS.md rule #9 ("No `any` types in TypeScript. Use proper types, generics, or `unknown` with type narrowing."). The actual shape consumed is small and stable (`reportNarrative`, `battlecard.{urgencyType, primaryEnterpriseLever, risks[]}`, `spend.{mtd, ytd, lifetime}`); the `skipKeys` array at line 537 is in effect a manual schema. A real interface would document the contract and let `tsc` catch the kind of typo this prompt builder is most exposed to.

## Polish (Detail)

### R4 — `as Parameters<typeof generateText>[0]` casts hide real bugs

Once the index signature `[key: string]: unknown` is removed from `GenerateTextParams` / `GenerateObjectParams` and the casts are dropped, the SDK's structural typing will catch issues like R1 at compile time. This is the single largest type-safety win in the repo.

### R5 — UTC "today" can misresolve OOO dates near day boundaries

`netlify/functions/extract-ooo-date.ts:58` anchors prompts to UTC. For users on PT, between ~16:00 and 23:59 the prompt's "today" is already tomorrow. If a member sends an OOO at 23:30 PT saying "back tomorrow", the model returns the day after they actually meant. Low-severity but easy to fix.

### R6 — Re-implemented constant-time compare

`netlify/functions/lib/auth.ts:3` rewrites `crypto.timingSafeEqual`. The codebase already uses the standard primitive in `identity-validate.ts`. Consolidating reduces surface area.

### R7 — `unknown` values flow into prompt template strings unfiltered

`netlify/functions/lib/prompts/reply.ts:32-33` reads `unknown` values from `promptConfig` and interpolates them into the system prompt. This is best-cased today (callers pass strings) but a non-string value would silently degrade to `[object Object]` inside the live prompt without any error path.

**Project overview:**
The project is a full-stack email automation system (Revenue Engine) for sales outreach. It uses React 19 on Netlify, an AWS Lambda + DynamoDB backend for workflow orchestration, and Netlify Functions (AI Gateway) to securely interface with LLMs (Claude, GPT, Gemini). The Netlify Functions handle text generation, reply classification, and thread summarization while shielding provider API keys from the Lambda environment.

**Architecture:**
- **`netlify/functions/`**: Independent serverless functions acting as specialized AI endpoints (`generate-email.ts`, `classify-reply.ts`, `summarize-thread.ts`, `extract-ooo-date.ts`).
- **`lib/ai-provider.ts`**: Provides multi-provider fallback logic (Anthropic -> OpenAI -> Google) using Vercel AI SDK and Netlify AI Gateway.
- **`lib/prompts/`**: Prompt builder strategy pattern that maps specific `promptType` strings to specialized prompt generation logic.

**Execution flows:**
I traced the email generation flow:
1. `services/api/src/functions/email/generate.ts` triggers a POST request to `NETLIFY_GENERATE_URL`.
2. The `generate-email.ts` Netlify function reads the JSON body and selects the appropriate prompt builder (e.g., `outbound-workflow.ts`).
3. The prompt builder compiles the system and user messages using injected context.
4. `generateObjectWithFallback` in `ai-provider.ts` queries the primary LLM (Anthropic) and falls back to OpenAI or Google upon failure, applying structured output schemas via Zod.
5. `sendAiCallback` executes an asynchronous delivery of the completion back to the backend.

**Test coverage gaps:**
The `netlify/functions` directory relies on internal modules with heavy string-manipulation and fallback chains (like `ai-provider.ts`). There are no collocated unit tests for these critical AI functions, which would be beneficial for catching regressions in prompt logic or fallback state handling independent of the AWS backend tests.

**Conventions:**
The code heavily relies on TypeScript strict mode, Zod schemas, and structured error responses. However, Netlify Functions return simple JSON `{ error: "message" }` strings instead of the nested `{ error: { message, type, code } }` Hono pattern used in the main API layer.

## Review

I reviewed the most recent commits, which focused largely on refactoring the frontend contact and enrollment lists to use Mantine table skeletons and creating a full-page contact detail view. The recent changes are exceptionally clean, leveraging `useMemo` for efficient sorting/grouping, managing layout cleanly with `Grid.Col`, and gracefully managing loading/error states.

**Defects & Root Cause Analysis:**
- **Identity Webhook Bypass (`identity-validate.ts`):** A logic flaw exists where `getWebhookSecret()` evaluates to `undefined` if environment variables are missing. The conditional check uses a short-circuit `webhookSecret && !verifyWebhookSignature(...)`, meaning a missing secret silently bypasses signature verification and allows unauthorized signups.
- **Silent Callback Failures (`lib/callbacks.ts`):** The `sendAiCallback` function relies on `fetch` and a `try/catch` block to handle errors. However, `fetch` only throws on network-level failures (like DNS or timeouts). If the callback URL returns a `500 Internal Server Error`, the catch block is bypassed, leaving the failure unmonitored.

**Specific recommendations:**
- `netlify/functions/identity-validate.ts:64`: Change the logic to explicitly check `if (!webhookSecret) return Response.json({ error: 'Config missing' }, { status: 500 })` to enforce a fail-closed state.
- `netlify/functions/lib/callbacks.ts:24`: Add `if (!res.ok) throw new Error('Callback rejected')` directly after the `fetch` await.

## Improve

**Polish items:**
- **Native Cryptography:** The `lib/auth.ts` file implements a custom JS `constantTimeEqual` loop. While functional, it is vulnerable to JIT optimization discrepancies. It is recommended to use Node's native `crypto.timingSafeEqual` for enterprise-grade security.
- **Date Standardization:** In `extract-ooo-date.ts`, the code relies on `new Date().toISOString().split("T")[0]` to determine "today". Since Netlify functions run in UTC, this might lead to off-by-one errors for relative date extraction (like "next Monday") depending on the contact's actual timezone. Consider passing a timezone explicitly in the prompt generation payload.
- **Error Response Consistency:** The `netlify/functions` handlers generally return `{ error: string }`. To match the convention defined for the Hono backend in `AGENTS.md`, these functions should ideally align to `{ error: { message: string, type?: string, code?: string } }`.

## Items Considered And Rejected

- **`timeout` usage in `lib/ai-provider.ts`**: I investigated whether the `timeout` property passed to Vercel AI SDK's `generateObject` was valid. While historically `abortSignal: AbortSignal.timeout()` was the canonical approach, modern versions of the Vercel AI SDK support the `timeout` parameter natively, meaning the fallback chain will reliably trigger.
- **`threadHistory` fallback in `summarize-thread.ts`**: I investigated whether the loop over `threadHistory` might crash if objects omit the `role` field. It defaults safely to `'Prospect'` and handles undefined `step` or `subject` values using short-circuits safely.
- **`length` check on strings in `outbound-workflow.ts`**: `ad.battlecard.risks?.length > 0` might iterate over characters if `risks` is mistakenly passed as a string rather than an array. I rejected this as a defect because the data originates strictly from trusted internal backend APIs that rigorously enforce the array shape.

The backend is a Serverless Framework AWS stack with a Hono OpenAPI monolith, standalone Lambda handlers, DynamoDB entity helpers, Gmail API integrations, EventBridge/SQS scheduling, and Netlify Functions for AI Gateway calls.

Architecture: `frontend/src/router.tsx` owns authenticated routes; contact and enrollment screens use query hooks under `frontend/src/hooks/queries`. `services/api/src/app.ts` mounts Hono route groups, while route shims such as `services/api/src/routes/contacts.ts` call standalone handlers where older Lambda code remains. Netlify AI generation enters through `netlify/functions/generate-email.ts`, validates an internal secret, builds a prompt, calls the provider fallback layer, and optionally posts an async callback.

Execution flows traced:
- Contact detail: `/contacts/$contactEmail` decodes the route param, calls `useContactActivity`, renders summary badges and timeline tabs, and links enrollment rows to `/enrollments/$enrollmentId`.
- Contact activity API: the Hono route wraps `functions/contacts/activity.ts`, which loads the contact, suppression, enrollments, action records, tracking events, sorts the timeline newest-first, and returns a summary.
- Enrollment list: `EnrollmentsStub` parses URL search state, calls `useEnrollments` with status/senders/workflowIds/cursor/limit, then locally filters and sorts the returned page.

Conventions observed: API errors generally use structured `{ error: { message, type, code } }` in Hono routes, DynamoDB access flows through entity/helpers, frontend state is URL-driven for list filters, and newer UI work uses Mantine components directly instead of the removed boneyard skeleton wrapper.
