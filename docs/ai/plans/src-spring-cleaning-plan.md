# `nax` Source Spring-Cleaning Completion Notes

## Scope

This cleanup kept `nax` as one npm package and one CLI binary. It did not introduce pnpm workspaces, package boundaries, or a publish-layout change.

The work focused on co-locating related source files, removing stale pass-through wrappers, defining dashboard/runtime contracts, and enforcing import boundaries so the source tree does not drift back to a flat `src/` layout.

Top-level `src/*.js` is now intentionally small. Remaining top-level implementation files are either CLI composition support or cross-cutting domain helpers that were not forced into `core/` because they carry Agent Runner, GitHub comment, blob offload, or installed-skill semantics:

- `agent-run-results.js`
- `agent-runner-sync.js`
- `comment-markers.js`
- `graceful-run-state.js`
- `handoff-sources.js`
- `issue-groups.js`
- `prompt-offload.js`
- `skills.js`
- `types.js`

No compatibility pass-through wrappers remain for the moved CLI, workflow, storage, integration, core, or dashboard-adjacent modules. Internal tests import the canonical implementation paths.

## Boundary Rules

`scripts/check-import-direction.js` now enforces:

- `src` must not import `bin/nax.js`.
- `src/dashboard/api` must not import local runtime modules or Node-only local execution APIs.
- `src/dashboard/web/src` must not import Node builtins, dashboard API internals, workflows, or integrations.
- `src/core` must not import local storage, dashboard, integrations, `fs`, or `child_process`.
- New top-level `src/*.js` files are blocked unless explicitly added to the small inventory allowlist.

## Runtime Interfaces

Dashboard/runtime seams are represented as plain JSDoc interfaces in `src/storage/interfaces.js`, including:

- `WorkflowCatalog`
- `RunStore`
- `EventStore`
- `LiveRuns`
- `DashboardMutations`
- `ArtifactStore`
- `BlobStore`
- `AgentRunnerTransport`

These are still in-process interfaces, not workspace packages. They are intended to let the local filesystem-backed dashboard and future hosted storage-backed dashboard share the same API shape.

## Shared Contracts

Shared dashboard DTO and workflow contract types live under `src/contracts/`.

The dashboard frontend now re-exports those types from `src/dashboard/web/src/types.ts`, so server/CLI JSDoc and frontend TypeScript no longer maintain separate copies of the same API shapes.
